RULE REFERENCE: What Each Rule Checks
================================================================================

R1 - Hardcoded Hex Colors (§2.6)
    Pattern: Background="#..." or Foreground="#..." in AXAML
    Expected: Use Classes or style references (DarkStyles.axaml)
    Impact: Breaks style consistency; color tokens can't be updated globally

R2 - Inline FontSize (§3.1)
    Pattern: FontSize="14" directly on elements
    Expected: Use TextBlock.Classes like "title", "header", "form_label"
    Impact: Typography not managed centrally

R3 - Inline FontWeight (§3.1)
    Pattern: FontWeight="Bold" on TextBlocks or Buttons
    Expected: Use TextBlock.Classes
    Impact: Typography consistency broken

R4 - Named Brushes for Status (§2.6)
    Pattern: Foreground="Red", "Green", "Blue", "Azure" used inline
    Expected: Use Material Icons + ToolTip (§1.2); status by shape not color
    Impact: Color-based UX violates accessibility; no separation of concerns

R5 - Hardcoded Strings (§3.2)
    Pattern: Text="English word" without {Binding
    Expected: Text="{Binding StrXxx}" from Localization.resx
    Impact: Blocks pt-BR localization; maintenance nightmare

R6 - Button Classes (§4)
    Pattern: <Button> without Classes attribute
    Expected: Classes="dialog1", "operation", "subButton", etc.
    Impact: Buttons inconsistent; decision matrix not followed

R7 - Spacing Scale (§5.5)
    Pattern: Margin/Padding with values not in {2,3,5,10,15,20,30}
    Expected: Only use scale values or compounds thereof
    Impact: Visual rhythm broken; spacing not scannable

R8 - Acrylic Borders (§1.3)
    Pattern: Window content not wrapped in ExperimentalAcrylicBorder
    Expected: All edit windows use Digger+Black+0.65 acrylic
    Impact: Visual continuity lost; main window doesn't feel cohesive
    Result: PASSING - all major windows compliant ✓

R9 - TextBlock Classes (§3.1)
    Pattern: TextBlock without Classes attribute
    Expected: Use form_label, header2, header_detail, etc.
    Impact: Typography not managed; false positives on static labels
    Risk: HIGH false-positive rate (40-50%)

R10 - Button Icon+Text Anatomy (§4.3)
    Pattern: Button with text not from Str* binding, or missing icon
    Expected: StackPanel[MaterialIcon + TextBlock(Str*)]
    Impact: Button anatomy inconsistent; localization fails
    Combined with: R5, R6

================================================================================
LIMITATIONS & FALSE POSITIVE RISKS
================================================================================

HIGH RISK (>25% false positives):
  • R9 TextBlock Classes: Can't distinguish static labels from bound content
  • R5 Hardcoded strings: Some in converters/templates are acceptable
  • R7 Spacing: Multi-value margins hard to parse without tokenization

MEDIUM RISK (10-25%):
  • R6 Button Classes: Complex XAML structures may be missed by regex
  • R1 Colors: May not catch all hex patterns in nested styles
  • R4 Named brushes: Binding vs hardcoded ambiguous in some contexts

LOW RISK (<10%):
  • R8 Acrylic: Pattern highly specific; low false-positive rate
  • R2 FontSize: Rule is simple and well-isolated
  • R3 FontWeight: Similar to R2; low false-positive rate

NOT CHECKED (require semantic/visual analysis):
  • Icon sizes (20×20, 15×15, 25×25 per class)
  • Window sizing conformance (§5.4 table)
  • Dialog action row layout (exact §4.4 structure)
  • Color contrast ratios (WCAG AA on all surfaces)
  • Elevation hierarchy (nesting depth)
  • Responsive resize behavior
  • Run-time binding correctness
  • Accessibility attributes
